# Agent Profiles

Этот файл фиксирует постоянные профили суб-агентов для репозитория `Kiarche-Continuum-War`.
Он нужен для предсказуемой оркестрации: кто за что отвечает, когда запускать агентов параллельно, а когда последовательно.

## Обязательный планер

Любая задача сначала проходит через планер, даже если на первый взгляд она кажется простой.

### Минимальный triage перед стартом

Перед началом нужно быстро определить:

1. Какой реальный результат нужен.
2. Какие каталоги и подсистемы затронуты.
3. Чем задача будет проверяться.
4. Есть ли зависимости, риски или скрытая междоменность.
5. Нужны ли суб-агенты или задача остается локальной.

### Классификация задач

- `Small`: изменение локальное, затрагивает один домен, способ проверки очевиден.
- `Multi-step`: есть несколько этапов, несколько файловых зон или отдельная стадия интеграции.
- `Uncertain`: причина проблемы неясна, есть риск ложной простоты, или непонятно, как верифицировать результат.

### Что делать по классам

- `Small`: короткий локальный план и выполнение.
- `Multi-step`: явный пошаговый план до начала реализации.
- `Uncertain`: сначала исследование, потом нормальный план, потом реализация.

### Когда повышать задачу до полного плана

- Если затрагивается больше одного домена.
- Если нужно тестирование в нескольких формах.
- Если есть риск тронуть Unity и docs web app одновременно.
- Если есть пользовательские ассеты, markdown, изображения или контентная синхронизация.
- Если после первичного чтения задача выглядит сложнее, чем в формулировке пользователя.
- Если неочевидно, как доказать, что задача действительно завершена.

## Стандарт изоляции контекста

Чтобы агенты не конфликтовали по состоянию задачи, используем полный набор:

- `task_id`
- `context_scope`
- `context_snapshot`
- `handoff_contract`
- `context_version`

### 1. task_id

У каждой задачи должен быть единый идентификатор.

Пример:

```text
KCW-2026-04-07-001
```

Он используется в плане, итерациях, handoff и проверках.

### 2. context_scope

Нужно заранее определить:

- какие папки и файлы входят в задачу;
- какие зоны исключены;
- какой агент владеет какой частью.

Пример:

```text
task_id: KCW-2026-04-07-001
context_scope:
  owned:
    - KiarcheContinuumWar/Assets/Scripts/Pathfinding/
  excluded:
    - KiarcheContinuumWar/Assets/Scripts/UI/
    - documentation/kcwweb/src/
  owner:
    - Agent B
```

Если два агента хотят один и тот же файл, это не рабочая ситуация, а сигнал на перепланирование.

### 3. context_snapshot

Это краткий снимок задачи, который передается агенту вместо всего разговора.

Минимальный состав:

- цель;
- подтвержденные факты;
- ограничения;
- способ проверки;
- открытые вопросы.

Шаблон:

```text
context_snapshot:
  goal: ...
  facts:
    - ...
  constraints:
    - ...
  verification:
    - ...
  open_questions:
    - ...
```

### 4. handoff_contract

Каждая передача между агентами должна быть формализована.

Минимальный состав:

- что сделано;
- что не сделано;
- какие файлы изменены;
- что проверено;
- какие риски остались.

Шаблон:

```text
handoff_contract:
  completed:
    - ...
  remaining:
    - ...
  changed_files:
    - ...
  verification_status:
    - ...
  risks:
    - ...
```

### 5. context_version

Каждая новая итерация после review, теста или уточнения пользователя повышает версию контекста.

Пример:

```text
task_id: KCW-2026-04-07-001
context_version: v1
```

После найденной ошибки:

```text
task_id: KCW-2026-04-07-001
context_version: v2
```

Это защищает от ситуации, когда агент продолжает работать по устаревшему описанию.

## Правила применения стандарта

- Не запускать multi-agent execution без `task_id`.
- Не передавать задачу суб-агенту без `context_scope`.
- Не передавать задачу суб-агенту без `context_snapshot`.
- Не принимать работу от агента без `handoff_contract`.
- После каждой итерации review/test обновлять `context_version`.
- Если контекст сильно изменился, старый handoff считается устаревшим и должен быть пересобран.
- Для выбора формата задачи использовать модульные шаблоны из `memory/task_templates.md`.

## Постоянный набор

### Agent A — Unity Gameplay
- Зона ответственности: `KiarcheContinuumWar/Assets/Scripts/Units/`, `Managers/`, `Core/`.
- Типовые задачи: поведение юнитов, боевые механики, состояние игры, игровой цикл, логика менеджеров.
- Не брать: pathfinding, map generation, UI, camera, web docs.
- Запускать первым, если задача про механику, боевую систему, ИИ юнитов или core gameplay.

### Agent B — Unity Systems
- Зона ответственности: `KiarcheContinuumWar/Assets/Scripts/Map/`, `Pathfinding/`, `Pooling/`.
- Типовые задачи: сетка, навигация, flow fields, генерация карты, оптимизация вычислений, инфраструктурные системы.
- Не брать: UI, input, camera, narrative docs.
- Запускать первым, если задача про производительность, навигацию, спавн, карту или системную архитектуру Unity-части.

### Agent C — Unity UX
- Зона ответственности: `KiarcheContinuumWar/Assets/Scripts/UI/`, `InputSystem/`, `CameraSystem/`.
- Типовые задачи: HUD, панели, пользовательское взаимодействие, камера, управление, feedback.
- Не брать: pathfinding, unit combat internals, web docs.
- Запускать первым, если задача касается удобства игры, интерфейса, камерного поведения или схемы управления.

### Agent D — Web Docs
- Зона ответственности: `documentation/kcwweb/src/`.
- Типовые задачи: React-компоненты, страницы, CSS, маршруты, typed data, markdown rendering.
- Не брать: Unity runtime logic, lore outside docs-site integration.
- Запускать первым, если задача касается сайта документации, лендинга, контентной структуры SPA или фронтенд-оформления.

### Agent E — Lore And Writing
- Зона ответственности: `documentation/`, `ovault/`.
- Типовые задачи: лор, описания фракций, структурирование markdown, консистентность терминов, narrative docs.
- Не брать: production code changes in Unity or React app, кроме мелких контентных правок для интеграции.
- Запускать первым, если задача про лор, тексты, иерархию документации или контентную синхронизацию.

### Agent F — Git
- Зона ответственности: git-состояние репозитория, логические группы изменений, diff, staging strategy, release/readiness.
- Типовые задачи: подготовка коммитов, анализ diff, проверка грязного дерева, безопасная работа с ветками, release handoff.
- Не брать: основную продуктовую реализацию, кроме минимальных git-related файлов и release metadata.
- Запускать после основных агентов, когда нужно привести изменения к чистому и понятному git-состоянию.

### Agent G — Dev-Ops
- Зона ответственности: deploy-артефакты и инструкции для `documentation/kcwweb/` и связанных landing/static deployment сценариев.
- Типовые задачи: подготовка VPS-friendly deploy scripts, env templates, run/build/restart commands, инструкции "скачай из репозитория и выполни команду".
- Не брать: gameplay logic Unity, lore editing, произвольные серверные изменения вне задачи деплоя.
- Запускать, когда нужно подготовить воспроизводимый деплой сайта или лендинга на личный VPS.

### Agent H — Review And Verification
- Зона ответственности: read-only проверка любой части репозитория.
- Типовые задачи: code review, поиск регрессий, проверка рисков, анализ diff, предложения по тестированию, follow-up по `npm run build` и `npm run lint`.
- Не брать: основную реализацию, если нет явного поручения.
- Запускать после основных агентов, когда нужен контроль качества или независимая проверка.

## Базовые сценарии запуска

### 1. Unity gameplay feature
- Основной агент: Agent A.
- Дополнительно: Agent F, если изменение затрагивает несколько подсистем или несет риск регрессии.

### 2. Pathfinding, map, pooling, performance
- Основной агент: Agent B.
- Дополнительно: Agent F для независимой проверки риска.

### 3. UI, input, camera
- Основной агент: Agent C.
- Дополнительно: Agent A, если UI меняет игровую логику.

### 4. Docs web app
- Основной агент: Agent D.
- Дополнительно: Agent H после существенных изменений структуры, стилей или typed data.

### 5. Lore or content work
- Основной агент: Agent E.
- Дополнительно: Agent D, если контент сразу встраивается в docs web app.

### 6. Git and release readiness
- Основной агент: Agent F.
- Дополнительно: профильный доменный агент, если нужно понять смысл изменений перед группировкой коммитов или release handoff.

### 7. VPS deployment for site or landing
- Основной агент: Agent G.
- Дополнительно: Agent D, если одновременно меняется код docs-site или landing.
- Дополнительно: Agent F, если нужно подготовить чистую release-поставку или release instructions по git-состоянию.

### 8. Cross-project task
- Параллельно:
- Agent A/B/C для Unity-части по домену.
- Agent D для docs web app.
- Agent E для narrative/docs части, если она независима.
- Agent G для deploy-подготовки, если задача включает VPS rollout.
- После этого локальная интеграция и при необходимости Agent F или Agent H.

## Правила параллельности

- Запускай агентов параллельно, только если у них разные каталоги владения.
- Не давай двум worker-агентам один и тот же файл или одну и ту же папку без жесткого разделения ответственности.
- Если задача зависит от результатов исследования, сначала explorer/review pass, потом worker.
- Если следующая реализация целиком зависит от архитектурного вывода, работай последовательно.

## Правила постановки задач суб-агентам

- Всегда указывай конкретные файлы или каталоги, которыми агент владеет.
- Всегда говори агенту, что он не один в кодовой базе и не должен откатывать чужие изменения.
- Формулируй ожидаемый результат явно: исследование, патч, review findings, verification summary.
- Если агент делает кодовые изменения, проси его перечислить измененные файлы в ответе.
- Если задача исследовательская, проси вывод в виде кратких, проверяемых тезисов, а не длинного эссе.

## Рекомендуемые комбинации

### Быстрый Unity-фиче-флоу
1. Agent A или B или C по домену.
2. Локальная интеграция.
3. Agent H для review.

### Большая кросс-доменная задача
1. Agent A/B/C по Unity-домену.
2. Agent D по docs-site.
3. Agent E по контенту.
4. Локальная сборка результатов.
5. Agent G, если нужен deploy-путь на VPS.
6. Agent F для git/release подготовки.
7. Agent H для итоговой проверки.

### Исследование перед реализацией
1. Agent H как explorer/reviewer или локальный анализ.
2. Профильный worker-агент.
3. Повторный Agent H при необходимости.

### Подготовка сайта к VPS-деплою
1. Agent D подготавливает или обновляет код сайта/лендинга.
2. Agent G готовит deploy scripts, env template и пошаговый rollout с одной командой запуска.
3. Agent F приводит итог в чистое git-состояние для передачи или релиза.
4. Agent H проверяет риски, build/lint и полноту инструкции.

## Цикл тестирования и доработки

Используй единый цикл:

1. Профильный агент делает реализацию.
2. Agent H или локальная проверка делает тест/review.
3. Если найдена ошибка, задача возвращается профильному агенту на доработку.
4. После доработки повторяется проверка.

### Лимит итераций

- Нормальный режим: до 3 итераций на одну и ту же проблему.
- Если после 3 полных циклов проблема не решена, автономный цикл останавливается.
- После этого нужно выйти к пользователю с кратким отчетом:
- что уже пробовали;
- какие симптомы остались;
- что именно блокирует уверенное решение;
- какие варианты следующего шага есть.

### Кому возвращать ошибку

- Ошибка в gameplay logic: обратно Agent A.
- Ошибка в pathfinding/map/pooling: обратно Agent B.
- Ошибка в UI/input/camera: обратно Agent C.
- Ошибка в docs web app: обратно Agent D.
- Ошибка в markdown/lore/content structure: обратно Agent E.
- Ошибка в deploy scripts/VPS rollout: обратно Agent G.
- Ошибка в git/release preparation: обратно Agent F.
- Если непонятно, где источник дефекта, сначала Agent H или локальный анализ на изоляцию причины.

## Практическое правило

Если задача маленькая, срочная или завязана на один файл, не делегируй.
Если задача распадается на независимые домены, делегируй параллельно.
Если один шаг определяет следующий, делегируй последовательно.
