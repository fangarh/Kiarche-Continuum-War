# VPS Deploy Workflow

Этот файл задает нормативный workflow для подготовки деплоя сайта и лендинга на личный VPS.

## Platform Baseline

- локальная разработка: Windows 11
- production VPS: Ubuntu 24
- синхронизация: Git

Это означает:

- production deploy path должен считаться Ubuntu-first;
- основной production deploy должен выполняться на VPS после получения актуального состояния репозитория;
- Windows используется для разработки, коммитов, пуша и локальной верификации, но не как основной production deploy executor.

Цель:

- чтобы нужные файлы лежали в репозитории;
- чтобы пользователь мог скачать актуальную версию;
- чтобы после одной понятной команды сайт был задеплоен или обновлен;
- чтобы деплой был воспроизводимым, а не "ручной магией".

## Область применения

Этот workflow относится к:

- `documentation/kcwweb/`
- возможным landing/static assets, если они живут в том же deployment-контуре
- deploy scripts и env templates, созданным для VPS rollout

Не относится к:

- Unity build/deploy
- произвольной серверной админке вне задачи выкладки сайта/лендинга

## Основной принцип

`Dev-Ops` агент не должен готовить уникальный ручной процесс под один сервер.
Он должен готовить:

- файловую структуру deploy-артефактов в репозитории;
- шаблон переменных окружения;
- скрипт или команду для воспроизводимого деплоя;
- краткую инструкцию для пользователя;
- ожидания по директориям на VPS.

Итог должен быть таким, чтобы пользователь мог:

1. скачать или обновить репозиторий;
2. заполнить env;
3. выполнить одну основную команду;
4. получить развернутый сайт.

## Целевой результат от Dev-Ops агента

Когда задача формулируется как "подготовь деплой на VPS", результат должен включать:

1. `deploy`-скрипт или набор скриптов в репозитории.
2. `.env.example` или иной шаблон конфигурации.
3. краткую инструкцию:
   - откуда запускать;
   - какие переменные заполнить;
   - какую команду выполнить;
   - что считается успешным деплоем.
4. описание структуры на VPS:
   - куда копируется build;
   - где лежит текущая версия;
   - нужен ли web server reload/restart.
5. способ отката или минимальную recovery-инструкцию, если деплой не удался.

## Рекомендуемая структура в репозитории

Если в задаче появляется VPS deployment path, предпочтительно использовать такую структуру:

```text
documentation/
  kcwweb/
  deploy/
    README.md
    .env.example
    deploy.ps1
    deploy.sh
    artifacts/
```

Допустимы и другие структуры, если они явно проще, но логика должна сохраняться:

- deploy assets лежат отдельно от app source;
- инструкция лежит рядом;
- переменные окружения не зашиваются в код;
- деплой можно повторить без ручной сборки знаний заново.

## Предпочтительная модель деплоя

Для текущего `kcwweb` как Vite static app базовый сценарий такой:

1. Локально или на VPS выполняется production build.
2. Артефакт `dist/` публикуется в целевую директорию сайта.
3. Web server отдает статические файлы из publish-директории.

Предпочтение по подходам:

### Вариант A — build на VPS

Использовать, если на сервере допустимо держать Node.js и npm.

Поток:

1. сервер получает репозиторий;
2. выполняется `npm install`;
3. выполняется `npm run build`;
4. `dist/` публикуется в web root;
5. при необходимости выполняется reload web server.

Плюс:

- проще единая команда на сервере.

Минус:

- VPS должен иметь корректную Node runtime.

### Вариант B — upload готового build

Использовать, если хочется минимальной зависимости VPS от Node toolchain.

Поток:

1. build создается локально;
2. готовый `dist/` копируется на VPS;
3. содержимое публикуется в web root;
4. при необходимости делается reload web server.

Плюс:

- меньше требований к серверу.

Минус:

- нужен более явный шаг упаковки/загрузки build-артефакта.

## Предпочтение по умолчанию

Если пользователь явно не попросил иное:

- для простого личного VPS допустим `Вариант A`, если нужен максимально прямой workflow;
- если важна предсказуемость окружения и минимум зависимостей на сервере, лучше `Вариант B`.

`Dev-Ops` агент должен явно указать, какой вариант выбран и почему.

## Git Sync Baseline

Базовый expected flow для этого репозитория:

1. разработка и проверка идут локально на Windows 11;
2. изменения публикуются в Git;
3. на Ubuntu 24 VPS выполняется обновление репозитория;
4. deploy запускается на VPS одной основной командой.

Предпочтительный рабочий поток:

```text
Windows 11 -> git push -> Ubuntu 24 VPS -> git pull -> deploy.sh
```

Если не оговорено иное, именно этот поток считается каноническим.

## Что должно быть в инструкции

Любой deploy workflow должен давать пользователю короткую инструкцию в таком формате:

```text
1. Открой каталог ...
2. Скопируй .env.example в .env и заполни ...
3. Выполни команду: ...
4. Проверь, что сайт доступен по ...
```

Если команда не одна, workflow не достиг нужной простоты и должен быть упрощен либо явно объяснен.

## Требования к "одной команде"

Под "одной командой" понимается одна основная команда запуска deploy flow.

Допустимо:

- один `powershell`/`bash` скрипт;
- один `npm run deploy`;
- один `docker compose up -d` для строго определенного deployment-контура.

Недопустимо:

- длинная последовательность из 6-10 ручных шагов без wrapper-скрипта;
- скрытые ручные действия, не описанные в инструкции;
- требование пользователю каждый раз вручную править пути внутри скрипта.

Для текущего репозитория при Ubuntu-first модели приоритет отдается `bash`-скрипту как production entrypoint.
PowerShell-скрипт может существовать как вспомогательный инструмент, но не должен считаться основным production deploy path без явного запроса пользователя.

## Что должен проверить Review агент

`Review And Verification` агент должен проверить:

- команда deploy действительно соответствует описанию;
- env-переменные перечислены явно;
- путь публикации описан;
- build path и deploy path не перепутаны;
- инструкция не зависит от неявного знания автора;
- есть минимальный rollback или recovery path.

## Роль Git агента в deploy workflow

`Git` агент подключается, если нужно:

- подготовить clean release state;
- выделить deploy-related изменения в понятную группу;
- убедиться, что deploy scripts и docs входят в релиз;
- подготовить handoff для публикации или release tag.

## Минимальный handoff для VPS deploy

Если задача передается пользователю или следующему агенту, handoff должен включать:

```text
deployment_notes:
  - deployment mode: build-on-vps | upload-dist
  - entry command: ...
  - env file: ...
  - publish path: ...
  - reload step: ...
  - rollback note: ...
```

## Антипаттерны

- Деплой, зависящий от устного знания автора.
- Скрипт без `.env.example` или иной явной конфигурации.
- Инструкция без указания рабочей директории.
- Деплой без объяснения, откуда берется production build.
- Смешивание app-кода и deploy-логики без структуры.
- Подготовка деплоя без финальной проверки `build`-пути.

## Практический стандарт для этого репозитория

Для `documentation/kcwweb/` Dev-Ops задача считается завершенной, если:

1. выбран deployment mode;
2. добавлены или обновлены deploy-артефакты в репозитории;
3. есть env template;
4. есть одна основная deploy-команда;
5. есть краткая инструкция для пользователя;
6. handoff содержит `deployment_notes`;
7. при необходимости Git-агент подготовил release-ready состояние.

Для текущей платформенной базы основной командой должна считаться Ubuntu/Linux-совместимая команда деплоя.
